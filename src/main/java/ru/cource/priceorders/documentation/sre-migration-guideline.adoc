== SRE гайдлайн при переезде на YC

1. SLA и отказоустойчивость
* Зафиксировать целевые уровни доступности для каждого сервисного слоя и проверить соответствие заявленной цели 99.995
для API Gateway и PG фактическим SLA YC, поскольку гарантия доступности различается по сервисам и публикуется отдельно
от общего SLA.
* Учесть многозонное развертывание как предпосылку к высокой доступности БД/сервисов, поскольку практики HA для Postgres
подразумевают масштабирование по зонам доступности для достижения целевых SLA на чтение/запись.
2. Нюансы репликации
* Логическая репликация поддерживается Managed PostgreSQL, но требует тождественности схем между источником и приемником;
схему следует переносить отдельно перед включением подписок.
* Нужен отдельный пользователь с ролью для логической репликации и корректными правами, иначе подписки не поднимутся и
восстановление не будет полным.
* Sequences не реплицируются логически, поэтому без синхронизации возможны коллизии ID при переключении и откате;
это критическая особенность blue‑green по Postgres.
* Каждая база потребует логический `replication slot`; при росте числа баз возрастает накладная нагрузка, что нужно
учитывать в `sizing`.
* Нужен контроль параметров логического декодирования и памяти соединений (например, `logical_decoding_work_mem`),
чтобы избежать застойных лагов при большом потоке WAL.
3. Cutover и откат
* Формализовать пошаговый runbook: проверить лаг, перевести учётки в `NOLOGIN`, завершить сессии на blue, переключить
маршрутизацию, выполнить верификационные запросы по публикациям/подпискам/слотам и `wal_level` перед релизом и для
быстрой обратной репликации при откате.
* Добавить явную процедуру синхронизации sequences в момент cutover и при rollback, иначе при двунаправленной работе
после отката возможны дубли/пропуски.
4. Бэкапы и PITR
* Подтвердить политику PITR: в Managed PG PITR включен по умолчанию с хранением WAL и дефолтным сроком жизни бэкапов,
настраиваемым при создании/редактировании кластера; это база для сценариев «горячего» отката.
* Протестировать восстановление кластера: в YC при ресторе создается новый кластер, а скорость восстановления зависит от
числа ядер (средняя оценка 10 МБ/с на ядро), что важно для RTO при авариях.
* Понимать механику архивации WAL и ее лаг: архив логов поступает в объектное хранилище с некоторой задержкой, что
влияет на «последнюю точку» при PITR.
5. Квоты и IOPS
* Проверить квоты и лимиты Managed PG по числу кластеров, CPU/RAM/хранилищу, соединениям и классам хостов, чтобы
целевая топология (монолит отдельно, микросервисы отдельно) и пиковая нагрузка укладывались в допустимые рамки.
* Для уменьшения риска IOPS‑throttling у Managed PG следует учитывать рекомендацию увеличивать объем хранилища как
способ поднять доступные IOPS/пропускную способность под нагрузкой монолита на NVMe (local SSD).
6. Interconnect и сеть
* Зафиксировать характеристики и резервирование Interconnect: это приватные выделенные соединения между площадками,
но их SLA/доступность следует связать с общими SLA YC и продумать дублирование каналов.
* С учетом выбранной схемы «приложение и БД в одном провайдере» Interconnect остается критичен для межсервисных
вызовов/репликации и должен быть протестирован на пропускную способность и latency под фоновыми нагрузками.
7. API Gateway
* Подтвердить требования к аутентификации/авторизации и возможности сервиса: API Gateway поддерживает JWT‑авторизацию
и авторизацию через сервисы, что влияет на дизайн отказоустойчивости и маршрутизации.
* Учесть квоты и лимиты API Gateway: размеры запросов/ответов, таймауты обработки, число доменов, что важно для планов
«один IP/маршрутизатор L7» и для оценки влияния пиков.
* Цель по доступности 99.995 для шлюза нужно проверить с учетом действующих уровней обслуживания по сервису.
8. Мониторинг и алерты
* Добавить мониторинг лагов репликации/слотов и проверок публикаций/подписок как SLO‑метрики cutover‑готовности согласно
рекомендуемым запросам проверки лагов/слотов.
* Мониторить WAL‑поток, IOPS и вероятность throttling, чтобы заранее реагировать на рост нагрузки и корректировать объем
хранилища/класс хостов.
* Учесть поведение реплик под синхронными/асинхронными гарантиями, так как чтения на репликах могут быть несогласованными
с мастерами при определенных режимах.
9. Пулы соединений
* Если используется пулер соединений, учесть, что логические потоки репликации проходят через Odyssey, что упрощает
топологию, но требует учета лимитов соединений и TTL.
10. Расширения и версии
* Проверить список поддерживаемых расширений и необходимость `shared_preload_libraries`; включение библиотек вызывает
рестарт мастера, что влияет на окна работ.
* Согласовать целевые версии PostgreSQL/расширений между Selectel и YC до начала логической репликации, поскольку
несоответствие версий/расширений часто нарушает совместимость схемы.
11. DDL и freeze‑окна
* Ввести freeze‑окно на DDL при включенной логической репликации, потому что корректность логической миграции зависит
от идентичности схем; изменения выполнять через управляемый экспорт/импорт схемы.
* Спланировать порядок доменных «групп переезда» так, чтобы не нарушать требования к публикациям/подпискам и
минимизировать необходимость временных DDL‑исключений.
12. Что еще уточнить
* Проверить multi‑AZ для Managed PG кластеров и соответствие её целевым ожиданиям по доступности чтения/записи
в эксплуатационных нагрузках.
* Составить регламент аварийных тренировок: частота проверки восстановления из бэкапов и контрольные метрики RTO/RPO с
учетом особенностей PITR и скорости восстановления в YC.
* Договориться о границах ответственности внутри команды SRE.
