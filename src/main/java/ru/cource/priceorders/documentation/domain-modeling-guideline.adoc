Гайдлайн для разработчика по моделированию в DDD с использованием EDA.

Базовая идея: каждый микросервис владеет собственной БД, взаимодействие — через доменные события по Kafka,
консистентность — через саги и transactional outbox/CDC, чтение ускоряется с помощью Redis cache‑aside,
а обработка событий делает упор на идемпотентность и транзакции Kafka EOS v2.

1. Начать с домена
* Зафиксировать Ubiquitous Language, определить Bounded Contexts и их границы ответственности, чтобы минимизировать
кросс‑контекстную связанность в дальнейшем.
* Для кросс‑контекстной интеграции заранее предусмотреть события и командные взаимодействия, избегая общего состояния
и общего доступа к таблицам.
2. Разбиение на сервисы
* Придерживаться паттерна Database per Service: у каждого сервиса собственная Postgres‑БД и схемы,
без шеринга таблиц между сервисами.
* Для операций, охватывающих несколько сервисов, отказаться от 2PC и использовать саги (оркестрация или хореография)
как стандартный механизм согласованности.
3. Агрегаты и транзакционные границы
* Проектировать агрегаты так, чтобы критичные инварианты (правила) выполнялись одной локальной транзакцией БД сервиса,
а внешние эффекты запускались событиями.
* Для сложных чтений (поиски, большие JOINы) использовать CQRS: командная модель в сервисах, а чтения — через
специализированные представления с асинхронным обновлением.
4. События и контракты
* Моделировать доменные события в терминах бизнес‑языка и версионировать их схемы,
выделяя `type/aggregateType/aggregateId/payload/metadata` для маршрутизации и эволюции.
* Выбирать в качестве ключа в сообщениях Kafka `aggregateId` для упорядоченности событий одного агрегата в рамках партиции.
5. Надёжная публикация: Outbox + CDC
* Писать событие сначала в таблицу outbox в той же локальной транзакции, что и изменение агрегата
(паттерн Transactional Outbox), исключая разрыв «БД обновили — событие не отправили».
* Реплицировать из outbox в Kafka через Debezium Outbox Event Router/CDC вместо собственных поллеров,
получая упорядоченное, надёжное и недублирующее извлечение.
6. Согласованность между сервисами: Saga
* Сага = последовательность локальных транзакций с компенсирующими действиями при сбое; выбирать хореографию (событиями)
или оркестрацию (центральный оркестратор) под сложность домена.
* Отдельно проработать контрмеры от аномалий параллельности и повторных записей (идемпотентность команд/событий).
7. Транзакции и EOS (exactly once semantics) в Kafka
* Включать Kafka EOS v2 в Spring Kafka: транзакционная запись в выходные топики в рамках одной операции.
* Понимать границы EOS: read→process→write «ровно один раз» в рамках Kafka, но при внешних системах (БД и др.) всё равно
нужна идемпотентность и согласованное участие в транзакции/откате.
8. Обработка и идемпотентность
* Реализовать идемпотентность потребителей: хранить обработанные  `messageId/ключи`, использовать `upsert/unique`‑ключи,
дедупликацию по `id`, и повторно безопасно применять операции.
* Конфигурировать retry с бэкоффом и Dead‑Letter Topic (DLT) на уровне потребителей для нерешаемых (бизнес) ошибок,
сохраняя порядок и идентификаторы.
9. Чтения и кэш Redis (cache‑aside)
* Встраивать Redis через Spring Cache как cache‑aside: сначала проверка кэша, при промахе чтение из Postgres и запись
в кэш с TTL/политикой инвалидации.
* Инвалидировать/обновлять ключи кэша при записях, избегая расхождений и используя подходы write‑through или явную
инвалидацию в зависимости от SLA.
10. Синхронизация и блокировки
* Для редких межэкземплярных критических секций можно применять распределённые блокировки в Redis с продлением lease и
осторожной настройкой таймаутов.
* Учитывать риски локов на Redis; при высокой критичности предпочитать проектные альтернативы (serial key) и строгую
идемпотентность.
11. Топики Kafka и «шина данных»
* Стандартизовать топики по доменным типам и версиям событий, привязать ключ к доменному идентификатору агрегата для
упорядоченности и масштабирования по партициям.
* Для межконтекстной интеграции строить EDA вокруг Kafka как «шины»: публикуем факты из источника истины, а потребители
строят свои проекции независимо.
12. Паттерны таблиц и событий
* Outbox в Postgres: `id` (UUID), `aggregateType` (varchar), `aggregateId` (varchar), `type` (varchar), `payload` (JSONB),
созданные в одной транзакции с доменной записью.
* Ключ сообщения в Kafka = `aggregateId`, а `aggregateType/type` помогают маршрутизации и приоритизации обработчиков.
13. Тесты и эволюция
* Покрывать саги контрактными и интеграционными тестами: happy‑path, компенсации, повторы, сбои на границах транзакций
и при ретраях.
* Эволюция событий — через совместимые изменения схемы и версионирование, поддерживая потребителей старых версий до
завершения миграции.
14. Чек‑лист внедрения
* Выделить контексты и агрегаты, зафиксировать события, определить, где нужны саги, и где достаточно eventual consistency.
* Включить transactional outbox + Debezium, настроить Spring Kafka EOS v2, реализовать идемпотентность и DLT/ретраи,
добавить Redis cache‑aside на горячие чтения.
15. Когда CQRS отдельно
* Выделять отдельные «read‑модели» и материализованные представления, если запросы сложные или с высокими SLA по
отзывчивости, держа их в синхроне событиями из Kafka.
* При необходимости — дополнительные индексы и кэши, не влияющие на командные транзакционные инварианты агрегатов.

Быстрый старт по шагам:

* границы и агрегаты,
* события и outbox‑таблица,
* Debezium в Kafka,
* потребители с EOS и идемпотентностью,
* саги для кросс‑сервисных операций,
* Redis cache‑aside на чтениях,
* ретраи/DLT/наблюдаемость.
