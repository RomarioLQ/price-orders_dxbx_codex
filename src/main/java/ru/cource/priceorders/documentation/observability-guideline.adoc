= Гайдлайн по техническому мониторингу сервисов
:toc:
:toclevels: 3

== Цель
Определить общие инженерные практики по работе с логами, метриками и распределенными трассировками, которые помогут:

* быстрее находить и устранять проблемы в работе продукта;
* формировать прозрачную картину работы сервисов;
* облегчить развитие и сопровождение продукта.

Таким образом, мы стремимся не просто мониторить известные проблемы, а иметь возможность исследовать и диагностировать неизвестные.

=== 0. Философия 

Данный гайдлайн не предписывает жесткие SLA/SLO. Вместо этого он дает командам свод принципов и рекомендаций для определения своих собственных целей. Конкретные цифры (например, целевая latency 200мс по p99) должны быть определены каждой командой самостоятельно.


== 1. Логирование

Логи — это хронологические записи событий, произошедших в системе. Их основная цель — дебаг и аудит.

=== 1.1 Принципы
* Минимально достаточные логи: логируем только то, что нужно для анализа; ни больше ни меньше.
* Логи должны быть структурированными (JSON/ключ-значение).
* Не логируем чувствительные данные (персональные данные, пароли, токены).
* Корреляция по trace‑id

=== 1.2 Уровни
Использовать уровни логирования (DEBUG, INFO, WARN, ERROR) по назначению

* *DEBUG* – подробности для отладки. Использовать только локально или точечно И временно в проде. По умолчанию уровень DEBUG отключен на проде.
* *INFO* – бизнес-события и ключевые изменения состояния системы. Важно: Не логируем каждый шаг алгоритма/процесса, потому что это создает огромный объем шума.
+
[NOTE]
====
В первые 2 недели после запуска новой фичи допускается повышенное логирование INFO.
Далее — оставляйте только ключевые события (если их кол-во имеет разумное кол-во) ИЛИ включаем семплирование.
====

Пример настройки семплирования для пропуска 10% INFO логов:

в pom.xml:
// Maven dependency для Janino
[source,xml]
----
<dependency>
    <groupId>org.codehaus.janino</groupId>
    <artifactId>janino</artifactId>
    <version>3.1.12</version>
</dependency>
----

в logback.xml добавляем кастомный фильтр:
// Logback appender с семплированием INFO-логов (~10%)
[source,xml]
----
<appender name="stdout" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
        <pattern>${STD_PATTERN}</pattern>
    </encoder>

    <!-- Фильтр для семплирования INFO-логов -->
    <filter class="ch.qos.logback.core.filter.EvaluatorFilter">
        <evaluator class="ch.qos.logback.classic.boolex.JaninoEventEvaluator">
            <expression><![CDATA[
                import ch.qos.logback.classic.Level;

                // Всегда пропускаем WARN и ERROR логи
                if (event.getLevel().isGreaterOrEqual(Level.WARN)) return true;

                // Семплируем INFO-логи с вероятностью 10%
                if (event.getLevel() == Level.INFO) return Math.random() < 0.1;

                // Отклоняем все остальное (DEBUG, TRACE)
                return false;
            ]]></expression>
        </evaluator>
        <OnMatch>ACCEPT</OnMatch>
        <OnMismatch>DENY</OnMismatch>
    </filter>
</appender>
----


* *WARN* – потенциальные проблемы, которые не ломают работу, но требуют внимания. Ошибка является частью workflow. Например "Not found product info in CRPT by GTIN list: [04660128370346]" или "Exist old session for lpPublicId 5b3c27c6-cbf5-11ee-9f0e-5bccd87e7cb3" или "Накладная идентична последней версии с внутренним номером DX006443A" (разумеется в зависимости от контекста).
* *ERROR* – ошибки, влияющие на достижение пользователем ценности или приводящие к деградации системы в т.ч. непредвиденные ошибки. Например, NullPointerException

[NOTE]
====
WARN::
Требует мониторинга и исследования.
ERROR::
Требует реакции СТО или ОР (кладем задачу в беклог или хотфиксим).

При прочих равных указывать ERROR

====

=== 1.3 Рекомендации
* Не спамить логами. Объем логов контролировать, при необходимости включать семплирование. Даже для крупных сервисов, как правило, N > 1млн логов/день тумач
* По максимуму разделять WARN от ERROR, хотя разница между ними может быть неочевидной. Помним о притче с мальчиком пастухом, который любил кричать "Волки, волки!". 
* Не используйте логи для сбора метрик. Для метрик... применяем метрики
* Если есть реальная необходимость хранить логи (например HTTP-response), то можно эти объёмы лить в свое S3 и позже грепать/анализировать. Важно: не лить в общее хранилище.

== 2. Метрики

Метрики — это агрегированные числовые данные о работе системы за определенный промежуток времени. Их цель — мониторинг производительности и трендов, а также генерация алертов.

=== 2.1 Принципы
* Метрика должна помогать ответить: *как понять, что сервис работает плохо?*
* Снимать не только средние значения (p50, p90, p99), но и `max`.
* Должны быть технические метрики уровня пользователя (latency API, error rate) и уровня системы (CPU, storage, Disk I/O).
* Могут быть бизнесовые метрики (см. категории ниже).
* Важно не только собирать метрики, но и реагировать на них: для критичных показателей должны быть настроены *алерты*

=== 2.2 Основные категории
* *Доступность и отзывчивость*: latency p50, p95, p99, max (API, DB), error rate.
* *Ресурсы*: CPU, RAM, storage, cache.
** Отдельно отмечу метрики объёма хранения данных И скорость роста объёма. 
* *Пропускная способность*: RPS, лаг консьюмеров, кол-во обращений к БД, отзывчивость конкретных запросов в БД (в мс).
* *Бизнес-метрики*: скорость обработки задач, кол-во ключевых событий.

=== 2.3 Рекомендации
* Максимальное значение `max` критически важно. Нередки ситуации, когда даже по p99 имеем целевое значение, но у нескольких клиентов(западло в том, что это в основном ключевые клиенты) что-то отваливается по timeout, при этом у большинства всё работает штатно. Тут на помощь приходят трассировки по `max`. 
* Выделите ключевые метрики для мониторинга И регулярно отслеживаете тренды
* Сравнивайте свои метрики с сервисами других команд. Если у вашего небольшого сервиса кол-во обращений к БД сопоставимо с кол-вом в монолитной БД – это звоночек 🔔

== 3. Распределённые трассировки

Трассировка — это сквозная запись пути одного запроса через распределенную систему. Ее цель — понимание workflow и диагностика проблем производительности.

=== 3.1 Принципы
* В трассировку должны попадать: вызовы БД, сторонние API, очереди, кеши.
* Трассировка должна показывать bottlenecks, циклы вызовов и проблему N+1.
* Каждый запрос должен иметь уникальный trace-id, проходящий через все сервисы.

### 3.2 Рекомендации
* Используйте инструмент link:https://apm.dxbx.ru/[Uptrace]. Трейсы линкуются, метрики агрегируются, логи подвязываются.
* Анализируйте трассировки на ретроспективах инцидентов.
* Длинные паузы (idle time) между span'ами: Указывают на то, что поток заблокирован (ожидание I/O, блокировка в коде, ожидание свободного connection pool).
– Семплируйте при большом кол-ве генерируемых span'ов (трассировки это про хайлоуд)

== 4. Командные практики

=== 4.1 Руководство к действию
* Анализируйте логи. Хлам не пишем. Полезное пишем, но умеренно
* Определите набор критичных метрик и SLO, ориентируясь на специфику продукта
* Внесите пункт "Наблюдаемость" в Definition of Done (DoD) для каждой фичи.
* Регулярно исследуйте трассировки по `max`

=== 4.2 Стартовые пороги для наблюдения

Ниже представлены значения для базовых метрик. Пороги усреднены и сформированы относительно монолита. Подразумевается, что у команд значения для сервисов должны быть получше.

База данных:

* latency: p50 < 3мс, p90 < 9мс, p99 < 25мс
* Скорость роста объёма БД менее 3% в месяц
* RPM: кол-во обращений в БД менее 100 000 в минуту
* Использование connection pool < 80%

HTTP запросы:

* latency: p50 < 30мс, p90 < 100мс, p99 < 300мс

Логи:

* RPS: кол-во логов менее 50 сообщений в секунду


== 5. Быстрый старт по шагам
* DEBUG не для прода
* INFO с ограничениями
* WARN != ERROR
* Latency по перцентилям
* Мониторим ресурсы
* Внимание на `max`
* bottlenecks и N+1 в link:https://apm.dxbx.ru/[Uptrace]
