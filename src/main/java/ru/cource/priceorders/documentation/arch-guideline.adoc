Архитектурный гайдлайн для бекенд‑разработчиков.

1. Цель
* Минимизировать простой и репутационные риски за счет надежной и простой архитектуры.
* Ускорять поставку фич и снижать стоимость сопровождения, удерживая рост инфраструктурных затрат при масштабировании.
2. Ключевые принципы
* Надежность превыше функционала. Фичи не стоят простоев. Есть бюджет ошибок: при выработке error budget приоритет
— улучшение надежности.
* Простота и иерархия важнее «гибкости». Предпочитаем простые, иерархические зависимости и отсутствие циклов.
«Все со всеми» — анти‑паттерн.
* Слабая связность, сильная сопряженность внутри границ. Компоненты минимум зависят друг от друга; внутри компонента код
однороден и понятен.
* Масштабируемость по горизонтали и деградация с сохранением ценности. Сервисы должны деградировать независимо, а не
"класть" систему целиком.
* Стоимостная осознанность. Любое решение оцениваем по влиянию на TCO (total cost of ownership): разработка,
сопровождение, инфраструктура (два последних 80% от стоимости).
* Обратная совместимость. Изменения не ломают клиентов; миграции по схеме expand–migrate–contract.
* Наблюдаемость по умолчанию. Без метрик, логов и трассировок нет эксплуатации, значит — нельзя в прод.
3. Модель декомпозиции и зависимости
* Архитектурные слои:
** Edge/Interfaces: API Gateway, BFF, backend, нет других способов интеграций.
** Domain/Product services: бизнес‑логика и данные конкретного домена.
** Foundation/Platform: аутентификация, биллинг, платежи, спейсы, нотификации, очереди, хранилища.
* Правило направленности: зависимости идут только вниз по слоям. Запрещены циклы и «боковые» вызовы между доменами без
явного контракта.
* Границы доменов: используем DDD‑подход (bounded contexts). Между контекстами — явные контракты (API/события) и
anti‑corruption layer при интеграции.
* Общая БД — запрет. Избегаем общей СУБД. Каждый сервис владеет своей схемой. Доступ к чужим данным — через публичный
API/события.
* Стратегия разбиения: начинаем с модульного монолита. Выделяем сервис, только когда есть устойчивые границы и
операционные выгоды (независимые релизы, масштабирование, авторство, SLA).
4. Контракты и версионирование
* Стабильные публичные контракты, версионирование SemVer. Backward‑compatible изменения предпочтительны.
* События и схемы в реестре контрактов (Backstage, OpenAPI). Контракт‑тесты обязательны.
* Изменения контрактов проходят архитектурный обзор и совместимы минимум N+2 минорных версий.
5. Синхронное vs асинхронное взаимодействие
* Синхронные вызовы — только в критическом онлайновом пути. Требуются: таймауты, дедлайны, ретраи с джиттером,
circuit breaker.
* Кросс‑доменная коммуникация по умолчанию асинхронная (event‑driven, outbox, очереди).
Используем eventual consistency и идемпотентность.
* Fan‑out и интеграции с внешними системами — через очереди/шины; не блокировать UX.
6. Надежность и отказоустойчивость
* Целевые уровни: для критичных API не ниже 99.95% (или выше по бизнес‑классификации).
Устанавливаем SLI/SLO на доступность и отзывчивость.
* Базовые техники (обязательны в каждом сетевом взаимодействии):
** Таймауты и дедлайны сквозные. Каждый downstream получает долю бюджетного времени запроса.
** Ретраи: максимум 2–3, экспоненциальная задержка + джиттер; только на безопасные ошибки и идемпотентные операции.
** Идемпотентность: ключи идемпотентности для POST, защиту от повторов обеспечиваем на приеме.
** Circuit breaker и bulkheads: изоляция пулов, ограничение параллелизма к зависимостям.
** Backpressure и load shedding: при перегрузке отбрасываем низкоприоритетные запросы, защищаем БД и очереди.
** Graceful degradation: кэш, заглушки, частичный функционал вместо 500.
* Состояние процесса:
** Readiness/Liveness пробы, постепенное завершение и дренирование соединений.
** Stateless‑сервисы, хранение сессий вне процесса.
* Данные и DR (data recovery):
** RTO/RPO(Recovery Time Objective/Recovery Point Objective) заданы по классам сервисов. Репликация БД, снапшоты,
тест восстановления.
** Multi‑AZ по умолчанию, межрегиональный DR для Tier‑1.
* Хаос‑инжиниринг: инъекция сбоев в staging, периодические учения для проверки сценариев отказа.
7. Производительность и масштабирование
* Бюджеты отзывчивости на запрос с распределением по слоям. Цели по P95/P99.
* Масштабирование — горизонтально сначала. Автоскейлинг по SLO‑релевантным метрикам (RPS, P95, очереди), а не только CPU.
* Пулы соединений и лимиты: защищаем БД/кэш. Для PostgreSQL — connection pooling (например, прокси‑пулер).
* Кэширование:
** Виды: in‑memory, распределенный, CDN. Явные TTL, инвалидация, защита от stampede (locking, jitter).
** Не кэшируем небезопасные или персональные данные без изоляции.
* Оптимизация запросов: индексы под частые фильтры, батчинг/пагинация, избегаем N+1.
* Пакетная обработка и очереди для тяжелых задач. Лимиты на размер партии и время.
8. Данные, целостность и согласованность
* Явные инварианты на уровне схемы (NOT NULL, FK там, где это не блокирует доступность), уникальные индексы.
* Миграции БД zero‑downtime: expand → backfill → switch → contract. Тщательные проверки планов/локов.
* Распределенные транзакции: избегаем 2PC. Используем саги/оркестрации и outbox‑паттерн.
* События — неизменяемые, со схемой, с версионированием. Повторная доставка допустима; потребители идемпотентны.
* Хранение: горячие/холодные данные, TTL и архивирование. ПИ соотносится с законами о данных и требованиями бизнеса.
9. CI/CD и управление изменениями
* Полностью автоматизированный конвейер: линтеры, статанализ, юнит/интеграционные/контрактные тесты, безопасность,
сбор и версионирование артефактов.
* Progressive delivery: feature‑флаги, canary/blue‑green, staged rollout, быстрый автобэкорлл на SLO‑события.
* Gate’ы перед продом: прохождение перформанс‑тестов и регрессии, наличие дашбордов и алертов, актуальные runbook’и.
* Совместимость вперед/назад — обязательна для деплоев без простоев.
10. Наблюдаемость и эксплуатация
* Логи: структурированные, корреляция по trace‑id; PII(personally identifiable info)‑политика.
* Метрики:
** RED для сервисов (Rate, Errors, Duration), USE для инфраструктуры (Utilization, Saturation, Errors).
** Бизнес‑SLI (конверсия запросов, успешные оплаты) — верхнеуровневые панели.
* Трейсинг end‑to‑end (distributed). Обязательна пропагация контекста.
* Алертинг: SLO‑driven, без шума, с приоритетами и четкими инструкциями.
* Runbook’и и чек‑листы на инциденты; постмортемы без поиска виновных, с действиями и владельцами.
11. Безопасность по умолчанию
* Аутентификация и авторизация централизованы (OIDC/OAuth2, сервис‑аккаунты). Политики минимальных привилегий.
* mTLS между сервисами, секреты в менеджере секретов. Регулярная ротация.
* Валидация входящих данных, защита от инъекций, ограничение размера/частоты запросов.
* Управление зависимостями: пинование версий, сканирование уязвимостей, SBOM(Software Bill of Materials).
* Аудит‑логирование критичных операций.
12. FinOps и контроль стоимости
* SLO(service-level objectives) по стоимости: целевая стоимость на 1k запросов/транзакций; отслеживание cost‑метрик
по сервисам/тенантам.
* Горизонталь вместо «золотых» машин. Профилирование, устранение горячих точек до апгрейдов железа.
* Выбор managed сервисов по TCO, а не по «быстрому старту». Стимулируем кэш/батч/асинхронность для удешевления.
* Осознанное хранение данных: классы хранения, TTL, компрессия.
* Регулярные cost‑ревью и архитектурные улучшения вместо бесконечного масштабирования инфраструктуры.
13. Работа с внешними зависимостями
* Обертки над внешними API: таймауты/ретраи/идемпотентность/CB(circuit braker), кэширование, деградация.
* Контроль квот и rate limits, разделение ключей по окружениям/тенантам.
* Мониторинг SLA поставщика как собственного риска; план B при отказе.
14. Процесс и управление архитектурой
* ADR на ключевые решения; RFC‑процесс перед изменением публичных контрактов и зависимостей.
* Архитектурное ревью: проверяем правила направленности зависимостей, надежность, данные, безопасность, стоимость.
* Каталог сервисов (Backstage): владелец, SLO, зависимости, дашборды, алерты, контакты on‑call.
* Инкрементальная эволюция: strangler‑pattern при переразбиении и избавлении от монолита.
15. Минимальные технические требования
* Таймауты и дедлайны заданы явно во всех HTTP клиентах, запросы отменяем по контексту.
* Для каждой точки записи — идемпотентность.
* Circuit breaker/bulkheads/backpressure включены для всех внешних вызовов.
* Readiness/Liveness, graceful shutdown во всех сервисах.
* Метрики RED, трассировка, структурные логи — до выхода в прод.
* Zero‑downtime миграции и стратегия отката.
* Нет прямых доступов к чужим БД. Контракты задокументированы и протестированы.
* Деплой только через CI/CD, фичи — под флагами, возможность быстрого rollback.
16. Чек‑лист перед выпуском сервиса/крупной фичи
* Определены SLI/SLO и error budget; дашборды и алерты готовы.
* Зависимости однонаправленные, без циклов; подтверждено инструментом (арх‑линтер).
* Контракты и схемы версионированы, есть контракт‑тесты.
* Настроены таймауты/ретраи/идемпотентность/CB/bulkheads.
* План миграции данных и отката; тест восстановления из бэкапа.
* Перформанс‑тесты на ожидаемую нагрузку + запас; проверены лимиты пулов/кэшей.
* План деградации и runbook для инцидента; on‑call уведомлен.
* Оценка стоимости: прогноз затрат после запуска и триггеры для оптимизации.
17. Смысл этих правил
* Снижая связность и упрощая иерархию, мы уменьшаем вероятность каскадных отказов и ускоряем изменения.
* Продуманная архитектура дает предсказуемое масштабирование и контролируемую стоимость, обеспечивая высокий уровень
сервиса без постоянного «доливать железо».
