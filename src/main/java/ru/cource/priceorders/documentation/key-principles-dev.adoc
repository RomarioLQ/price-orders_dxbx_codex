== Цель
Добиться **ПРОСТОТЫ и ГИБКОСТИ** кода и, как следствие, повысить надежность и скорость разработки,  
руководствуясь всего двумя ключевыми принципами:

* **Закон Деметры (Law of Demeter)** — минимизация избыточных связей между компонентами
* **SLAP (Single Level of Abstraction Principle)** — единый уровень абстракции внутри метода. Говоря простым языком – "Не смешивай бизнес-логику и технические детали в одном методе"

=== 1. Закон Деметры (Law of Demeter) 

Сервис или класс должен взаимодействовать **только со своими ближайшими зависимостями**  
(с теми объектами, которые он напрямую знает).

Иными словами:  
Не лезь вглубь чужих объектов и не инжектируй то, что уже инкапсулировано в другом сервисе.

Еще проще:
Если сервис `А` взаимодействует с `B`, а `B` взаимодейстует с `C`, то `A` не должен напрямую взаимодействовать и даже знать про `C`. Если `A` что-то нужно из `C`, то `A` просит об этом `B` (своего соседа/друга)

=== 1.1 Закон Деметры. Пример нарушения 

Разберем пример нарушения Закона Деметры на уровне зависимостей, который встречается при проектировании решений.

[source,java]
----

public class StoreFacade {
    private final ProductService productService;
    private final ProductFacade productFacade;
    ...
}
----
[source,java]
----
public class ProductFacade {
    private final ProductService productService;
    private final ProductConverter converter;
    ...

=== 1.2 Закон Деметры. К чему приводит нарушения

* Увеличивается связанность и когнитивная сложность кода
* Снижается устойчивость кода к изменениям. Эффект домино. При измении 1 сервиса требуется внести правки в N сервисов. Рефакторинг становится необъятным...
* Дублируется ответственность, нарушается `SRP`

Беспорядочные инъекции КРАТНО усложняют рефаторинг и тестирование кода.

=== 1.3 Закон Деметры. Рекомендации

* Каждый сервис/класс должен видеть и взаимодействовать только с соседними слоями, никаких прыжков через несколько слоёв
* Оценивай границы сервисов/классов. Проверяй зависимости и взаимодействия на предмет дублирования ответственности и нарушения инкапсуляции
* Доступ к Item имеет только корневой компонент. Нельзя инжектить условный InvoiceItemService! Инжектируй InvoiceService и пусть только в нём будет логика взаимодействия с айтемами
* Закон Деметры актуален и применим для любых взаимодействий, как на уровне кода, так и на уровне архитектуры

=== 2. SLAP 

В одном методе не нужно смешивать `что делаем` (бизнес-логику) и `как именно делаем` (технические детали). 

Иными словами:  
Метод должен читаться как **последовательная бизнес-история**, а не как смесь бизнес-логики, проверок и технических деталей.

=== 2.1 SLAP. Пример нарушения 

Разберем пример нарушения SLAP:

[source,java]
----

// ❌ В этом методе перемешаны уровни абстракции: получение и сохранение заказа, валидация, обработка ошибок, бизнес-логика сохрание в БД
    public void processOrder(Long orderId) {

        var orderOpt = orderService.findOrder(orderId);
        if (!order.isPresent()) {
            throw new OrderNotFoundException("Order not found");
        }

        var order = orderOpt.get();

        if (order.getStatus() == OrderStatus.CLOSED || order.getStatus() == OrderStatus.DONE ||
        order.getCreateDateTime().isBefore(LocalDateTime.now().minusHours(1))) {
            throw new OrderStateException("Order cannot be edited");
        }

        var discount = discountCalculator.calculateDiscountBy(order.getCustomerId());

        order.applyDiscount(discount);
        order.setState(OrderStatus.AWAITING_PAYMENT);

        orderService.save(order);
    }

----  

Метод читается "рвано", требует фокусировки внимания даже на незначительных деталях, а бизнес-смысл теряется. Теперь отрефакторим 🪄:

[source,java]
----
// ✅ Публичный метод читабелен и отражает бизнес-процесс "как историю"
    public void processOrder(Long orderId) {

        var order = getOrder(orderId);
        validate(order);
        var discount = calculateDiscount(order.getCustomerId());
        applyDiscount(order, discount);
        markAsAwaitingPayment(order);
        save(order);

    }

    private Order getOrder(Long orderId) {
        return orderService.findOrder(orderId).orElseThrow(() -> new NotFoundException());
    }

    private void validate(order) {
        if (isTerminated(order)) {
            throw new OrderStateException("Order cannot be edited");
        }

        if (isOrderExpired(order)) {
            throw new OrderExpiredException("Order expired");
        }
    }

    private boolean isTerminated(Order order) {
        // лучше даже сделать order.isTerminated(), что будет соответствовать принципу Tell, Don't Ask, 
        // но оставлю здесь для наглядности исходный вариант
        return order.getStatus() == OrderStatus.CLOSED || order.getStatus() == OrderStatus.DONE;
    }

    private boolean isOrderExpired(Order order) {
        return order.getCreateDateTime().isBefore(LocalDateTime.now().minusHours(1));
    }

    private BigDecimal calculateDiscount(Long customerId) {
        return discountCalculator.calculateDiscountBy(customerId);
    }

    private void applyDiscount(Order order, BigDecimal discount) {
        order.applyDiscount(discount);
    }

    private void markAsAwaitingPayment(Order order) {
        order.setState(OrderStatus.AWAITING_PAYMENT);
    }

    private void save(Order order) {
        orderService.save(order);
    }
----

Теперь метод `processOrder` читается **как последовательность шагов бизнес-логики**:
найти заказ → проверить заказ → рассчитать скидку → применить → изменить состояние → сохранить.

* Все технические детали вынесены в приватные методы (вероятно в часть из них вам даже не придется смотреть при повторном открытии метода `processOrder`) 
* Код приобрел свойство `самодокументируемости`. 
* Код стал проще для чтения, понимания и отладки. Привет👋 принцип `KISS`. Теперь код может понять не только разработчик, но и любой аналитик/тестировщик и легко сопоставить код с требованиями
* Каждый приватный метод говорит на том же языке домена (`заказ`, `скидка`, `ожидание оплаты`)

Может показаться, что кода стало больше, но на практике вы не будете постоянно погружаться в каждый приватный метод. Также необязательно любое действие, например `order.setState(OrderStatus.AWAITING_PAYMENT);` выносить в отдельный метод, хотя имхо спустя время код с методами читается и воспринимается легче.

=== 2.2 SLAP. К чему приводит нарушения принципа

* Нельзя быстро увидеть, “что делает метод”, не вчитываясь в “как он это делает”. В одном месте смешаны бизнес-логика, проверки и технические детали
* Код плохо переиспользуется — логика не разбита на самостоятельные операции. Полезные куски кода просто закопаны внутри больших методов, а значит высока вероятность дублирования → нарушается принцип `DRY` и `SRP`
* Код требует комментариев – нет свойства `самодокументируемости`
* Сложность поддержки. Приходится анализировать весь метод для мелких правок. Высокий риск внести ошибки
* Сложный код — как правило, нерабочий код. Отлаживать и ревьють крайне затруднительно и долго
* Сложный код — как правило, медленный код. Видеть узкие места и оптимизировать всё также крайне затруднительно и долго

Коротко: Нарушение SLAP → монолитные методы → технический долг → замедление разработки.

=== 2.3 SLAP. Рекомендации

* SLAP позволяет дизайнить решение даже на псевдоКоде. Почти as is можно интерпретировать требования: _найти заказ → проверить заказ → рассчитать скидку → применить скидку → изменить состояние → сохранить_
* С оговорками можно сказать, что SLAP это отображение(приземление) DDD, только на уровне кода 
* SLAP прекрасно сочетается и взаимоусиливает синергию с принципами `DRY`, `SRP`, `KISS`, `Explicit is Better Than Implicit`, `Tell, Don't Ask`
